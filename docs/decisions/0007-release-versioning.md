# 0007 - Release versioning and automated release notes

Status: accepted (2026-07-14). Implements #218. Owner call: automate the "should
we cut a release?" nudge rather than release fully by hand or fully hands-off.

## Context

The project had no version story: `package.json` was pinned at `0.0.0`, GitHub
Pages redeployed on every merge to `main`, and there was no changelog or tagged
release marking a coherent build. This is the first slice of the release and
delivery thread from the 2026-07-14 brainstorm (issue #218; the sibling #219
adds per-PR preview deploys).

The owner wanted the decision of *when* to release to stay theirs, but the
*noticing* that a release is due to be automatic, so they do not have to
remember or hand-run a sequence of commands.

## Decision

1. **Semantic Versioning** (`MAJOR.MINOR.PATCH`), driven by the Conventional
   Commit style already used in this repo. `feat:` bumps the minor, `fix:` /
   `perf:` bump the patch, and a `!` or `BREAKING CHANGE` bumps the major.

2. **release-please** (`googleapis/release-please-action@v4`) maintains a
   standing "release PR" on `main`. It reads the commit history and keeps that PR
   updated with the next version and a generated changelog section. The PR is the
   automated nudge: it tells the owner what a release would contain. Merging it
   cuts the release (creates the `vX.Y.Z` tag, publishes a GitHub Release, bumps
   `package.json`, and updates `CHANGELOG.md`); leaving it open costs nothing and
   it keeps updating as more commits land. This is the middle path the owner
   asked for: automatic detection, manual (one-click) decision.

3. **`CHANGELOG.md`** in Keep a Changelog format. A hand-written `0.1.0` baseline
   entry summarizes the state of the game when the process was introduced;
   everything from `0.2.0` onward is generated by release-please, so released
   sections are not edited by hand.

4. **Baseline anchoring.** The manifest records `0.1.0` as the last release and
   `bootstrap-sha` in the config pins the starting commit, so the first
   automated release PR only accounts for changes made *after* this setup (it
   proposes `0.2.0`), rather than regenerating a changelog from the entire prior
   history. This mirrors the issue-workflow rule of tracking from here forward
   instead of bulk-migrating old history.

5. **A non-default token for the release PR.** The workflow runs release-please
   with `secrets.RELEASE_PLEASE_TOKEN`, falling back to the default
   `GITHUB_TOKEN` when that secret is absent. GitHub deliberately does not raise
   `pull_request` events for actions taken by `GITHUB_TOKEN`, so a release PR it
   opens receives no CI run at all, and `main`'s required `check` and `e2e`
   status checks then block the merge forever. A fine-grained PAT (or a GitHub
   App installation token) with `contents: write` and `pull-requests: write`
   makes the PR author a real identity, so CI runs normally. See #238; this bit
   the 0.2.0 release, which needed a manual close/reopen to trigger CI.

## Consequences

- Adds one GitHub Action and two config files (`release-please-config.json`,
  `.release-please-manifest.json`). No npm/runtime dependency, so the Pages
  deployment is unaffected; the deploy workflow still runs on merge to `main`
  independently of releases.
- The release flow depends on a repo secret that lives outside the repo. If
  `RELEASE_PLEASE_TOKEN` is missing or expired, releases do not break loudly:
  they silently fall back to the old behaviour, where the release PR has no
  checks and needs a manual close/reopen to merge. A PAT with an expiry date is
  therefore a dated maintenance item; that is the price of not running a GitHub
  App for a solo project. The current PAT expires **2026-12-31**; rotation is
  tracked in #243.
- Tags and Releases now exist as durable markers a rollback or a "what shipped
  when" question can point at.
- Commit message discipline now has teeth: the changelog quality is exactly the
  commit-message quality, reinforcing the existing Conventional Commit rule.

## Alternatives considered

- **Manual tags** (`npm version` + push a tag, workflow publishes): simplest and
  no new machinery, but the owner would have to remember to do it. Rejected
  because it does not satisfy the "tell me when a release is due" goal.
- **Fully automated release on every merge**: maximal automation but removes the
  owner's control over release cadence and would tag noisy intermediate states.
  Rejected; the release-PR model keeps the human in the loop cheaply.
