## The autoplay finding

`scripts/autoplay.mjs` had no npm script, no README entry, and no CI job, so
nothing pointed at it and nobody ran it. It was broken three ways:

1. **Machine-pinned paths.** `ROOT` hardcoded to one checkout; output written to a
   dead session scratchpad under `/tmp`.
2. **Stranded at 0/52 in Greybridge after 4 deliveries.** The driver never
   repairs, so once the travel sink (#134) landed it wore to 0 and limped at
   0.15x. Same limp-crawl face PR #137 fixed for `full-arc.spec.ts` via `?nowear`;
   autoplay never got it.
3. **Never bought an upgrade.** Since #161 replaced "B buys cheapest" with a
   menu, `B` opens it and number keys buy while open. The driver still assumed the
   old behaviour, so it fitted zero upgrades and finished on **3357 idle coins**.
   The menu it left open also made `handleBoardInput()` swallow every first
   contract-accept press.

Defect 3 is the one that matters. That log reproduces the false flatline signal
that opened the roads-gate thread (2563 idle coins, no upgrades) - a thread since
DELIVERED and validated by human play at a 576c capstone. **A tool nobody ran was
manufacturing the symptom.** Anyone reading its output would have reopened settled
work.

Verified by running it, which is the only thing that would have caught any of it:
capstone reached, 23 deliveries, all 7 upgrades fitted, no runtime errors, against
stranding at 4 before.

## Decisions

- **The memory store is empty by design, not merely tidy.** The failure mode is
  not the writing, it is that nothing ever deletes. Every entry looked worth
  keeping the day it was written; a rich cache goes stale and then misleads with
  the confidence it once informed. Four of the 20 would have sent a session to
  rebuild closed work.
- **Salience belongs in `CLAUDE.md`, not memory.** The last four pointers added no
  content (all their facts were in the repo); they added a nudge to read
  something. A nudge that exists on one machine only is worthless to the owner's
  other subscription. `CLAUDE.md` loads everywhere, so that is the mechanism that
  actually transfers.
- **autoplay kept, not dropped.** The case for dropping was real (no consumer, and
  its numbers had misled once). It was kept because its job is distinct from the
  other two drivers and the owner values this category of work. Note the
  first recommendation to keep it rested on a wrong premise: that PR #245 touching
  it meant it was maintained. #245 was a mechanical base-path sweep. Running it
  would have shown that in one minute, and did.
- **autoplay runs `?nowear`, so the coins in its log are flow, not economics.**
  There are no repair bills. `travel-sink-measure.spec.ts` owns economics. Teaching
  autoplay to repair would substantially duplicate that spec and was declined.
