## Completed

### Diagnosis (settled)

- The failing check is the **Arc workflow**, which only runs post-merge on `main`
  (gated behind `RUN_ARC`, deliberately skipped on PRs). So #132 merged on
  `check` + `e2e` and the Arc first ran minutes later, red. #134 inherited it.
- Arc was **green through #129** and broke at **#132**. But #132 touched only
  data/missions/dialogue/save, nothing in movement, and `regionCleared()` already
  counted the base contracts it "added" to the mission spine. So #132 did not
  change the driver's behaviour; this is a **latent harness fragility that CI
  load newly exposed** (the #107 / #114 / #121 flake lineage).
- Confirmed by reruns: deterministic-ish red on CI, never reproduces locally.

### Root cause (two faces, both load-only)

1. **Coast past home.** When a drive ends the wagon coasts a variable number of
   tiles past the home settlement. A post-coast `atHome` read then reports false,
   so the accept/talk window is missed. The stall watchdog signature is
   position-blind (by design, to catch aimless driving), so it reads the
   oscillation one tile off home as a stall and throws after 60 steps. Locally the
   frame loop is fast, the coast is tiny, the wagon settles on home.
2. **Travel-sink limp-crawl.** `limpMultiplier` is binary: full speed until
   condition hits 0, then 0.15x. On a loaded runner a long leg can drain the
   wagon mid-drive, so `driveToTile` cannot crawl the rest within its step budget
   and throws "courier stuck (x,y)".

### PR #137 (branch `fix/arc-coast-past-home`)

- `c3afe8f` - one-tile coast tolerance + re-seat before the accept key; added
  `workflow_dispatch` to `arc.yml`. CI: 1 green, then 1 red (revealed more faces).
- `f64bbaa` - always re-seat home (no fixed tile threshold); repair (R) at every
  settlement stop; `driveOrRecover` tow-and-repair on a stranded stuck; step
  budget 250 -> 400. CI: **1 green on first push**, then a rerun **failed on the
  480s timeout (8.7m)** because the per-settlement repairs roughly doubled the
  drive, plus another coast stall in fenmarch.

Local full arc is green on both commits (3.8m / 4.3m). `tsc` and eslint clean.

## Next actions

Recommended lean approach (removes the timeout and a whole failure class):

1. **Disable wagon wear in the arc via a `?nowear` e2e flag.** The Arc is a
   reachability / soft-lock guard, not a travel-sink test (the sink is unit-tested
   and cannot soft-lock: a dry wagon still limps to a settlement and a rescue is
   always available). Concretely:
   - `src/scenes/map-scene.ts`: add `wearDisabled()` gated on `?e2e` + `?nowear`,
     and skip only the condition-mutation lines in `trackDistance` when it is set
     (keep the trip-distance and `tilesSinceAccept` tracking).
   - `tests/e2e/drive.ts`: have `bootE2E` append `nowear` (fold into the turbo
     boot or add a `noWear` option).
   - `tests/e2e/full-arc.spec.ts`: boot with it. This lets you **revert the
     per-settlement repairs and re-seats from `f64bbaa`** (they caused the 8.7m
     timeout). Keep always-re-seat-home and `driveOrRecover` (now a harmless
     safety net). Raise `test.setTimeout(480_000)` -> `720_000` for headroom.
2. **Validate with several CI samples** (not one - it is a flake). Use the temp
   push trigger on the branch, or `workflow_dispatch` once it is on `main`. Want
   3-4 consecutive green.
3. **Remove the temporary branch line** from `arc.yml`'s push trigger before merge.
4. **Arm auto-merge** (`--squash --auto`) once green and clean.

Alternative: this is exactly issue **#121** (demote the Arc now that reachability
