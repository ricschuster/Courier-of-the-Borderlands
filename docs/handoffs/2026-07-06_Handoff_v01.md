# Session Handoff: 2026-07-06

## Summary

Cleared the handoff v04 feature backlog (three new e2e tests plus the optional
settings cleanup), then reworked the region topology into a hub with
arrival-at-marker travel, and moved the Fenmarch gateway to a different side of
the hub map. Everything shipped as small CI-gated PRs (#23 to #25) and is on

## Completed

E2E coverage and cleanup (PR #23):
- Extracted the input-drive helpers from `playthrough.spec.ts` into a shared
  `tests/e2e/drive.ts` (`readTick`, `driveToTile` with an `onTile` callback,
  `bootE2E`, `collectErrors`, and later `pressUntil` / `travelTo`).
- `upgrade.spec.ts`: complete a delivery, drive home, buy the cheapest upgrade
  (`far-lantern`, 40c) with the B key; assert coins spent and upgrade persisted.
- `ford-route.spec.ts`: prove the ford unlock changes pathfinding, not just a
  flag. Pre-unlock the route to the east bank detours over the bridge; post
  unlock it routes through the ford and is strictly shorter; then drive across.
- `fenmarch-unlock.spec.ts`: mirror the Greybridge ford-unlock test in Fenmarch,
  booting from a seeded save.
- `?e2e` hook gained owned `upgrades` and a full-path `pathTo(x,y)` helper.
- Pruned the duplicated permission allowlist from `.claude/settings.local.json`
  (now just machine-specific `env.PATH`); moved the machine-agnostic entries
  into the checked-in `settings.json`.
- Fixed a CI flake: single Phaser keypresses can drop on a throttled frame under
  parallel load, so `pressUntil` re-presses the accept (`1`) and buy (`B`)
  inputs until they register. Both are no-ops once complete.

Hub region layout and arrival markers (PR #24):
- Topology changed from a linear chain to a hub. Greybridge is the hub with a
  gateway to each spoke; Saltreach and Fenmarch each link only back to the hub.
  The direct Saltreach <-> Fenmarch link is gone.
- Travel now lands the courier on the return gateway (the marker back to where
  it came from), not the region's spawn. Pure `arrivalTile(region, fromRegionId?)`
  helper in `region-system.ts`; fresh load with no origin still uses spawn.
- `map-scene.ts`: `MapSceneData.fromRegionId`; place courier via `arrivalTile`;
  expose `gateways` on the `?e2e` hook (hook version bumped to 3).
- `tests/e2e/travel.spec.ts` drives a real Greybridge <-> Saltreach round trip
  and asserts the landing tile each way (new `travelTo` helper). Hub topology
  and `arrivalTile` are unit tested. ADR `docs/decisions/0003-...` added;
  `docs/design/03_regions.md` updated.

Fenmarch gateway placement (PR #25):
- Moved the Greybridge -> Fenmarch gateway from the east edge (19,10) to the
  south edge (13,10), so the two spokes leave the hub on different sides (east
  to Saltreach, south over the bridge to Fenmarch). Extended `travel.spec.ts`
  with a Greybridge <-> Fenmarch round trip that verifies the moved tile is
  reachable and arrives correctly.

Notes for the next session:
- Maps currently fit the viewport (20x11 at 48px = 960x528), so there is no
  camera follow yet. That is the keystone unlock for larger maps (see below).
- Fog is saved per region as revealed tile indices. Changing a shipped region's
  dimensions invalidates its saved fog indices, so a size change needs a fog
  migration (or invalidate that region's fog on a dimension change).

## Next actions (feature backlog)

Theme for next session: bigger, more explorable maps. Build one per branch/PR,
CI-gated. Suggested order:

1. Camera follow + minimap adjustment. Add a camera that follows the courier
   (world bounds are already set in `create()`); adapt the minimap (currently
   6px/tile from a corner) so it still works when the map exceeds the viewport.
   Self-contained rendering change; do this first, as it unblocks any map larger
   than the screen.
2. Seeded PRNG utility. Small foundational helper. Also makes weather selection
   (currently `Math.random`) deterministic and testable. Useful on its own.
3. Property-based region invariant tests: every settlement reachable from home,
   every contract's pickup -> destination route exists, the ford is genuinely
   shorter than its alternative. Hardens the hand-authored maps now.
4. Expand one region to roughly 2x size (with the camera in place). Playtest the
   loop feel and tune content density (more settlements/terrain variety so the
   larger map does not read as empty) before scaling further. Watch fog rect
   count and add a fog migration for the resized region.

Decisions already made (do not re-litigate):
- Randomly generated maps: rejected. Fully random runtime maps fight design
  pillar 4 (story through places) by losing authored settlement notes, route
  names, and tone. If procedural generation is ever wanted, do it as an offline
  authoring tool that produces candidates to hand-curate into authored regions,
  not a runtime feature. Not on the backlog.

Not in the auto-queue (need human input): audio and juice (asset/licensing) and
an art pass beyond grey-box.

## Risks or decisions needed

- Camera work is the gate for both bigger maps and any future region that does
  not fit on screen. Decide the target map size before authoring content for it.
- Whether to keep authored maps only, or invest in the offline generation tool,
  once hand-authoring more/larger regions becomes the bottleneck.
- Still open from prior handoffs: whether to flip `enforce_admins` on branch
  protection, and whether to keep the Node v24 `env.PATH` pin in
  `settings.local.json`.
