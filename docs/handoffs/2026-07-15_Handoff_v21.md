## Telemetry: how it populates

The dashboard reads `localStorage` under `courier-of-the-borderlands/telemetry`
on whatever origin it is opened from. The game writes a record at exactly two
moments and nowhere else:

1. A region is cleared, `map-scene.ts` `captureTelemetry('region')`.
2. The arc capstone fires, `captureTelemetry('arc')`.

There is no per-delivery logging, so a short session records nothing. That is
why the page was blank: the feature only shipped 2026-07-14, and clearing a
whole region is the minimum event.

Three things worth knowing before reading any number off it:

- **`full-arc.spec.ts` runs with `noWear: true`.** Telemetry from it reports
  zero wear and zero strands: data that looks real and is false. Use
  `travel-sink-measure.spec.ts` (`MEASURE_DIFF=standard`), which runs wear ON
  and exists to measure the sink.
- **The dashboard has export and clear, but no import.** There is no UI path to
  load records back in; seeding needs a console `localStorage.setItem`.
- **A Playwright run's storage is discarded when it ends.** The records here
  were captured by temporarily dumping localStorage at the end of the measure
  spec, then reverting that edit. Nothing was committed.

Sample arc (standard, wear ON): 23 deliveries, 4 records, 318 total wear, min
condition 17, **0 strand events**. Bot numbers, not player numbers: the driver
routes near-optimally and repairs at every home visit, so its wear is a lower
bound and its condition an upper bound. See #264.

## Why GitHub started mailing constantly

Two features landed seven minutes apart on 2026-07-14 and together made every
merge generate mail. Neither is misconfigured.

- **PR previews (#234, merged 23:01:28).** The action comments once per PR. It
  edits that comment later, and edits do not notify, so it is one email per PR.
  Its first ever comment (`2026-07-14T22:59:10Z`, 15:59 PDT, the owner's first
  email) was on **its own PR**: `pull_request` workflows run from the PR's head
  branch, so #234 triggered the workflow it was adding, against itself.
- **release-please (#232, merged 22:54:55).** It keeps a standing release PR and
  rewrites its branch on every push to `main`. PR #240 carries 14
  `head_ref_force_pushed` events, which is the cadence of the `chore(main):`
  mail.

**This half self-resolved.** The force-push actor was `github-actions[bot]` only
because the token was broken. Since the PAT was set (03:58:52), release-please
acts as the owner, and GitHub does not mail you about your own actions by
default. The last two force-pushes are attributed to `ricschuster`.

Advice given, not yet applied: set the repo watch to "Participating and
@mentions", and unsubscribe from #240 (closing/reopening it through the owner's
account during the #238 workaround subscribed them to that thread). Future
release PRs will be authored by the owner via the PAT, so each release cycle
still costs about one email, the preview comment on the release PR.
