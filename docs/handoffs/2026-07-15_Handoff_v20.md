## The release token (#238), reopened

The workflow wiring landed, but there is no token behind it. `gh secret list`
shows `RELEASE_PLEASE_TOKEN` exists, which is what made this look done and got
#238 closed. The warning step added in #246 says otherwise on every run:

```
##[warning]RELEASE_PLEASE_TOKEN is unset or empty, so release-please is
falling back to GITHUB_TOKEN.
```

The secret was created with an empty value. This is exactly the trap the
workflow comment calls out: `gh secret set` with no TTY stores an empty value
and exits 0. The #246 warning did its job, turning a silent failure into a
visible one.

**Owner action:** run `gh secret set RELEASE_PLEASE_TOKEN` interactively and
paste a real PAT, then check the next release-please run logs
`RELEASE_PLEASE_TOKEN is set` instead of the warning. This also makes #243
(rotate before 2026-12-31) moot until a token with a real expiry exists.

## What the FAQ taught us

Writing player-facing answers forced every claim to be checked against the
code, and the docs lost twice (#255, #257). Both stale entries read perfectly
well; they just described a game that no longer existed. An early FAQ draft of
mine did the same thing in reverse, telling players to "drop the contract"
before travelling when no drop mechanic exists.

Worth remembering: doc review that only reads the docs cannot catch this. The
README review (#236) had already passed over the `G` bug.
