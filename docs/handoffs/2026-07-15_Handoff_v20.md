## The release token (#238): now fixed

**Resolved later the same session.** The section below records why it was
reopened; the owner then set a real PAT (03:58:52) and it is verified working,
so #238 is closed and nothing here is outstanding.

Proof came in two steps, because the first was not enough. The next run logged
`RELEASE_PLEASE_TOKEN is set`, but that only showed the secret had a value. The
real test was whether a release PR gets CI unaided, which needed the PAT to
push to the release branch. Landing #262 (a `fix:`) did it: release-please
pushed at 10:13:21, CI fired on its own, and release PR #240 went from
"no checks reported" and BLOCKED to all checks passing and CLEAN. Merging it
cuts 0.3.0 whenever wanted.

#261 added `workflow_dispatch`, so re-running release-please no longer needs a
dummy commit.

### Why it was reopened (kept for the record)

The workflow wiring landed, but there is no token behind it. `gh secret list`
shows `RELEASE_PLEASE_TOKEN` exists, which is what made this look done and got
#238 closed. The warning step added in #246 says otherwise on every run:

```
##[warning]RELEASE_PLEASE_TOKEN is unset or empty, so release-please is
falling back to GITHUB_TOKEN.
```

The secret was created with an empty value. This is exactly the trap the
workflow comment calls out: `gh secret set` with no TTY stores an empty value
and exits 0. The #246 warning did its job, turning a silent failure into a
visible one.

It was fixed by setting the secret through the GitHub web UI rather than the
CLI, which sidesteps the stdin trap above and avoids `--body` putting a live
token into shell history.
